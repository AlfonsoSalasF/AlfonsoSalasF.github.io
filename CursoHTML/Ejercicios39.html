Ejercicio 1: Ventajas de la Validación HTML.

Enumera al menos tres ventajas clave de validar tu código HTML utilizando el W3C Markup Validation Service.

Al menos tres ventajas clave de validar tu código HTML:

  1. Compatibilidad entre navegadores: Un HTML válido tiene más probabilidades de ser interpretado y renderizado de manera consistente por diferentes navegadores, reduciendo los problemas de visualización y "bugs" específicos de cada navegador.
  2. Accesibilidad mejorada: Un código HTML bien estructurado y validado es más fácil de ser comprendido por tecnologías de asistencia como los lectores de pantalla, haciendo que tu sitio sea más accesible para usuarios con discapacidades.
  3. Facilita la depuración: El validador señala errores y advertencias específicas, lo que acelera el proceso de encontrar y corregir problemas en tu código que de otra manera serían difíciles de diagnosticar.
  4. Mejora del SEO (indirectamente): Aunque la validación no es un factor directo importante para el SEO, un código limpio y semántico ayuda a los motores de búsqueda a rastrear e indexar tu contenido de manera más eficiente.
  5. Mantenimiento y colaboración: Un código válido y estructurado es más fácil de leer, entender y mantener por ti mismo en el futuro o por otros desarrolladores que trabajen en el mismo proyecto.

Ejercicio 2: Refactorización de Código (Indentación y Semántica).

El siguiente fragmento de código HTML tiene problemas de indentación y no utiliza etiquetas semánticas donde podría. Refactorízalo para mejorar su legibilidad y semántica.

<div>
<p>Mi Blog</p>
    <div>
        <a href="#">Home</a>
    <a href="#">About</a>
    <a href="#">Contact</a>
    </div>
    <div>
    <div>
        <p>Título del post</p>
            <p>Este es el contenido del post.</p>
        </div>
</div>
<div>
    <p>Copyright 2023</p>
</div>
</div>

Aquí tienes una versión refactorizada, mejorando la indentación y utilizando etiquetas semánticas:


<!-- Contenedor principal del sitio -->
<div class="site-wrapper">

    <!-- Encabezado principal del sitio -->
    <header class="site-header">
        <h1>Mi Blog</h1>
        <nav class="main-navigation">
            <ul>
                <li><a href="#">Home</a></li>
                <li><a href="#">About</a></li>
                <li><a href="#">Contact</a></li>
            </ul>
        </nav>
    </header>

    <!-- Contenido principal del blog -->
    <main class="blog-main-content">
        <!-- Un artículo/post del blog -->
        <article class="blog-post">
            <h2>Título del post</h2>
            <p>Este es el contenido del post.</p>
        </article>
    </main>

    <!-- Pie de página del sitio -->
    <footer class="site-footer">
        <p>&copy; Copyright 2023</p>
    </footer>

</div>

Ejercicio 3: Nomenclatura de Clases e IDs.

Para cada uno de los siguientes elementos, sugiere un nombre de clase o ID que siga las buenas prácticas (descriptivo, kebab-case, etc.) y explica brevemente por qué:

  1. Un <button> que envía un formulario.
  2. Una <div> que contiene la barra lateral de navegación.
  3. Un <input type="text"> para el nombre de usuario en un formulario de registro.
  4. El <section> principal que contiene toda la información de contacto de la página.
  1. <button> que envía un formulario.
    • Nombre sugerido: class="submit-button" o class="form-submit-btn"
    • Por qué: Descriptivo de su función, utiliza kebab-case y es fácilmente reusable para otros botones de envío.
  2. <div> que contiene la barra lateral de navegación.
    • Nombre sugerido: class="sidebar-navigation" o class="main-sidebar"
    • Por qué: Indica claramente que es una barra lateral y contiene navegación, o que es la barra lateral principal. Usa kebab-case.
  3. <input type="text"> para el nombre de usuario en un formulario de registro.
    • Nombre sugerido: id="username-input" o class="user-input" (si hay varios)
    • Por qué: Es específico del campo de usuario. Si es único en la página y se usará para JavaScript o un ancla, un id es apropiado. Si su estilo es genérico para inputs de usuario, una class es mejor. Ambos siguen el estilo de nombres claros.
  4. El <section> principal que contiene toda la información de contacto de la página.
    • Nombre sugerido: id="contact-section" o class="contact-info"
    • Por qué: Claramente identifica la sección como la de contacto. Un id es adecuado si solo hay una sección de contacto principal en la página (como suele ser el caso). Si pudiese haber varias, una class sería más apropiada.
